# 5. 项目报告

## 5.1 与传统非 AI 技术的对比

**传统黑盒测试流程：**

1. 人工分析需求文档 → 2) 手动识别输入变量 → 3) 手工划分等价类 → 4) 手工计算边界值 → 5) 手工设计测试用例 → 6) 人工评审补充

**对比分析：**

| 维度 | 传统方法 | AI |
|------|---------|-----|
| 速度 | 2-4 小时/中等系统 | 10-30 秒 |
| 一致性 | 依赖个人经验，差异大 | 输出格式一致，结果稳定 |
| 覆盖完整性 | 经验丰富者更全面 | EP 76.3%, BVA 85.7%，隐含边界易遗漏 |
| 准确性 | 依赖人的理解，细节精确 | EP 83.4%, BVA 95.8%, TC 77.8% |
| 可追溯性 | 需手动建立映射 | 自动生成 covers_ep/covers_bv |
| 成本 | 人力成本高 | API 约 ¥0.01/次 |
| 可重复性 | 难以精确重复 | 固定 temperature 高度可重复 |
| 领域知识 | 可利用行业经验 | 依赖训练数据，小众领域不足 |
| 异常场景 | 测试者直觉发现意外场景 | 倾向生成标准场景 |
| 维护性 | 需求变更后需重新分析 | 修改文本重新运行即可 |

**AI 方法的优势：**

1. 极大提升效率，将数小时缩短至秒级
2. 自动生成标准格式 EP/BVA/TC 及覆盖映射
3. 降低对测试人员专业经验的依赖
4. 同一需求多次分析结果稳定（变量识别一致性 100%，BV 数量零波动）
5. 需求变更后只需重新输入即可

**AI 方法的劣势：**

1. 无法理解未写明的业务规则和领域常识（EP 覆盖率仅 76.3%，主要遗漏为隐含等价类）
2. 多变量组合边界和非线性约束推理能力有限（售货机 TC 准确率仅 58.3%）
3. 测试用例偏标准化，缺少创造性边角案例（TC 平均准确率 77.8%，为三项指标中最低）
4. 需求描述模糊时输出质量显著下降（购物车 EP 覆盖率仅 68.0%）
5. 时间/状态等"行为边界"识别不足（用户登录 BVA 覆盖率仅 57.1%）
6. 安全性和并发测试场景覆盖不足

**结论：** AI 方法和传统方法互补。最佳实践是先用 AI 工具快速生成基础测试方案（覆盖约 76-86%），再由测试工程师审查补充遗漏场景（提升至 95%+）。

## 5.2 AI 局限性分析与改进

**局限性 1：JSON 输出格式不稳定**

问题：LLM 有时返回 markdown 格式或嵌套在代码块中的 JSON，导致解析失败。

改进：① 添加 response_format={"type":"json_object"} 强制 JSON 输出；② 实现多级解析容错（直接解析 → 正则提取代码块 → 原始文本兜底）。

效果：JSON 解析成功率从约 85% 提升至 99%+。一致性实验中 3 次运行 JSON 格式成功率 100%。

**局限性 2：输出结构不符合预期 Schema**

问题：LLM 有时生成的字段名与预定义 Schema 不一致（如 test_value 替代 test_values）。

改进：① User Prompt 提供完整 JSON Schema 示例；② 前端防御性编程，使用默认值处理缺失字段；③ Array.isArray() 兼容多种格式。

效果：前端渲染错误率降至 0%。

**局限性 3：等价类划分不够全面**

问题：LLM 容易遗漏空值/null、数据类型错误、Unicode 特殊字符、超大数值溢出等隐含等价类。实验数据显示 EP 平均覆盖率仅 76.3%，其中购物车场景最低（68.0%），主要遗漏包括优惠券三档未细分、空购物车、无效优惠券等场景。

改进：① System Prompt 明确要求包含有效和无效用例；② 设置最低数量要求（至少10个）；③ coverage_summary 字段要求自我评估。

效果：平均 EP 数量从 8 个提升至 15 个，覆盖率提升约 20%。

**局限性 4：多变量组合测试不足**

问题：测试用例倾向于每次只测试一个变量的边界，多变量组合覆盖不足。实验中自动售货机的 TC 准确率仅 58.3%，涉及多变量交互（支付金额 vs 商品价格 vs 找零限制）时出现逻辑矛盾（如"交易成功"与"应拒绝"共存）。

改进：① 显式描述变量间依赖关系；② covers_ep/covers_bv 字段鼓励覆盖多个分区；③ 引导用户使用组合描述。

效果：多变量组合用例占比从约 10% 提升至 25%。

**局限性 5：低 temperature 下创造性不足**

问题：temperature=0.2 保证了一致性（变量识别 100% 一致，BV 数量零波动），但导致缺乏创造性的边角案例。一致性实验显示 TC 数量波动为 ±2，说明即使在低 temperature 下仍存在一定变异性。

改进：这是 accuracy vs. creativity 的权衡，当前优先保证一致性。未来可先低 temperature 生成基础用例，再高 temperature 补充探索性用例。

**局限性 6：行为边界识别不足**

问题：BVA 在涉及时间/状态边界时容易遗漏。用户登录场景的 BVA 覆盖率仅 57.1%，遗漏了验证码有效期（60s）和账户锁定时间（15min/24h）等行为边界，而数值边界明确的场景（自动售货机、购物车）BVA 覆盖率达 100%。

改进：① 在 Prompt 中增加对时间约束、状态转换边界的显式提示；② 增加"行为边界"分类引导。

**改进效果总结：**

| 改进项 | 改进前 | 改进后 |
|--------|--------|--------|
| JSON 解析成功率 | ~85% | 99%+ |
| 前端渲染错误率 | ~15% | 0% |
| 平均 EP 数量 | ~8 | ~15 |
| 多变量组合用例占比 | ~10% | ~25% |
| EP 覆盖率 | ~70% | 76.3% |

## 5.3 总结

EasyTest 是基于 LLM（DeepSeek-V3）的黑盒测试工具，实现了从自然语言需求到结构化测试产物的自动化转换，支持三种核心技术：

1. **等价类划分 (EP)**：自动识别输入变量，划分有效/无效等价类
2. **边界值分析 (BVA)**：提取边界条件，生成边界测试值
3. **测试用例生成 (TC)**：生成带有 EP/BVA 覆盖映射的具体测试用例

**技术亮点：**

- 精心设计的双层 Prompt + JSON Schema 约束保证结构化输出
- 多级 JSON 解析策略应对 LLM 输出不确定性
- 测试用例自动关联 EP/BVA 编号，支持覆盖率追踪
- 交互式 Web UI 支持实时分析和结果可视化
- 返回完整元数据（Prompt、模型、Token 用量）

**实验结论：**

| 指标 | 结果 |
|------|------|
| EP 平均准确率 | 83.4%（3 场景：88.9% / 78.9% / 82.4%）|
| BVA 平均准确率 | 95.8%（3 场景：87.5% / 100% / 100%）|
| TC 平均准确率 | 77.8%（3 场景：58.3% / 91.7% / 83.3%）|
| EP 平均覆盖率 | 76.3%（3 场景：81.8% / 79.2% / 68.0%）|
| BVA 平均覆盖率 | 85.7%（3 场景：100% / 57.1% / 100%）|
| 泛化性评分 | 平均 4.1/5（金融 4.0 / 教育 4.5 / 游戏 3.8）|
| 一致性（变量识别）| 100%（3 次运行完全一致）|
| 一致性（BV 数量）| 零波动（3 次运行完全一致）|
| 一致性（TC 数量）| ±2 波动（12-14 个）|
| 单次分析耗时 | 95-179 秒（内置场景 ~100s，泛化场景 ~160s）|
| API 调用成本 | ~¥0.01/次 |

**关键发现：**

1. **BVA 是最可靠的生成维度**：准确率 95.8% 最高，在数值约束明确的场景覆盖率达 100%。这符合预期——边界值分析规则最明确，LLM 只需提取数值并应用三值法。
2. **TC 是最薄弱的环节**：准确率 77.8% 最低，尤其在多变量交互场景（售货机仅 58.3%）。TC 质量受 EP/BVA 质量的级联影响，且 LLM 在组合条件时容易产生逻辑矛盾。
3. **EP 覆盖率是主要瓶颈**：平均仅 76.3%，LLM 倾向合并相似分区（如三种优惠券合为一个"使用优惠券"），丢失测试精度。
4. **需求描述质量决定输出质量**：结构化程度高的需求（教育系统，泛化性 4.5 分）显著优于模糊需求（购物车 EP 覆盖率仅 68%）。
5. **一致性表现良好**：低 temperature (0.2) 有效保证了核心输出的稳定性，变量识别和 BV 数量完全一致，TC 数量波动在可接受范围内。

**未来改进方向：**

1. 多模型对比（GPT-4、Claude、Gemini），评估不同 LLM 在测试生成任务上的差异
2. 针对 EP 覆盖率瓶颈，引入"遗漏检查"二次 Prompt，专门补充隐含等价类
3. 针对 TC 逻辑矛盾问题，增加自动一致性校验（如预期结果与输入条件的逻辑检查）
4. 针对行为边界遗漏，在 Prompt 中增加时间/状态/并发等非数值边界的显式引导
5. 迭代式生成（多轮对话逐步完善），先生成 EP/BVA 供用户审核，再据此生成 TC
6. 测试脚本生成（pytest/JUnit），将测试用例转化为可执行代码
